## 2. R1-R5 detailed playbooks

### R1: `__match_any_sync` emulation perf on AMD/Intel

**Why it matters.** The Vulkan port must approximate CUDA encode throughput on AMD and Intel Arc to be a credible alternative to the CUDA backend. CUDA's `__match_any_sync` is one PTX instruction; the Vulkan emulation costs 11+ ballot iterations (site #3) or a 32-broadcast OR-reduction (site #1). If the emulation regresses encode throughput by >35% on AMD/Intel, the Vulkan port is shippable but uncompetitive for production use on those vendors, which threatens the "full vendor coverage" locked constraint.

**Detection.** M5 stands up a STANDALONE microbenchmark — port JUST the two match_any emulations (site #1 and site #3) as `.comp` shaders with no other kernel logic. M5 runs on the M8a minimal dispatch + timing chassis that was split out from the draft's M8 specifically so M5's perf numbers are physically obtainable (M8a depends only on M4; M5 depends on M4 + M8a). Run on every CI lane. Measure ns/dispatch on synthetic workloads sized to match production huff_build_lut (site #3) and lz_encode (site #1) shapes. Compare to CUDA reference baseline running the same workload sizes. **Pre-committed numeric threshold: AMD throughput ≥ 65% of CUDA on BOTH sites; same for Intel. UNTESTED-PASS IS DISALLOWED.**

**Playbook if triggered.**
1. Confirm the measurement: re-run on three different AMD/Intel devices in the CI matrix to rule out a one-off driver issue.
2. Profile the SPIR-V via `spirv-cross` and AMD/Intel native profilers (RGP / VTune Graphics) to identify the specific stall.
3. If the stall is in the 11-iter ballot loop (site #3): investigate `subgroupBallotBitCount` lowering on the failing vendor; possibly replace with a manual `bitfieldExtract` loop.
4. If the stall is in the 32-broadcast OR-reduction (site #1): promote `subgroupPartitionedXorNV`-equivalent fast path from "opportunistic" (NVIDIA only) to "required for Tier-1 AMD/Intel parity" — investigate whether `VK_AMD_shader_ballot` or `VK_INTEL_*_subgroup_*` extensions offer equivalent primitives.
5. If no fast path exists for AMD/Intel: file `streamlz-vk/issues/arch-revision-r1-{vendor}`, invoke cutline §7 — demote AMD/Intel to Tier-2 with documented status strings (`"AMD Tier-2 mode: ..."` / `"Intel Tier-2 mode: ..."`), ship v1.0-alpha with documented perf disclosure in README.
6. PAUSE M11 (first kernel port) until decision is documented in `docs/vulkan_port_architecture.md` §7.
7. If M5 hardware is unavailable (Reviewer 3 fatal): UNTESTED-pass is NOT an option — either acquire real AMD + real Intel hardware OR explicitly defer R1 retirement and downgrade M22a/M22b acceptance to NVIDIA-only with AMD/Intel marked `pending real hardware in v1.0-final`. Cutline §7 reflects the deferral; v1.0-alpha ships NVIDIA-Tier-1-only.

**Why we measure now and not later.** If the standalone measurement at M5 is fine but a later production-kernel measurement (M18/M22b) shows worse perf, the problem is integration overhead (other code in the kernel), not emulation cost — and is fixable inside the consuming kernel without re-architecting. If the standalone measurement at M5 is below threshold, the problem is fundamental and warrants pausing.

### R2: `subgroupSize=32` unsupportable on mobile

**Why it matters.** Tier-1 requires `subgroupSize=32` pinned via `VK_EXT_subgroup_size_control`. If a Tier-1 whitelisted vendor (e.g., a future AMD driver, or an Intel Arc revision) silently fails to honor the pin at runtime, Tier-1 kernels execute with wrong lane semantics and produce incorrect bytes. Mali is hardcoded to Tier-2 (no Mali gen advertises 32 in min/maxSubgroupSize), but Adreno is more uncertain.

**Detection — split into two phases.** Architecture §22 calls R2 "subgroupSize=32 unsupportable on mobile means Tier-2 emulation has to actually work" — the classification portion (probe correctly classifies the device) is testable at M4; the emulation-correctness portion (Tier-2 actually produces byte-identical output on real mobile silicon) is only testable when real mobile production kernels run, which is M29. The draft's claim that M4 alone retired R2 was overclaim. R2 is split:
- **M4 retires R2 classification**: probe runs at every `slzCreate_vk` call; vendor whitelist enforced. **M4 also runs the probe on at least one real mobile borrow** (Pixel 8 Adreno or Pixel 6 Mali via adb-attached CI worker) to validate the whitelist.
- **M29 retires R2 emulation correctness**: real Adreno + real Mali run production decode + encode kernels with the documented reduced corpus (4 cases × L1..L5 — NOT the draft's L1..L3, which never exercised the L5→L3 clamp path).

**Playbook if triggered.**
1. Reproduce on the failing device — capture `vkGetPhysicalDeviceProperties2` output for the device.
2. Check whether the device supports `VK_EXT_subgroup_size_control` with `subgroupSizeControl=VK_TRUE` AND the range includes 32. If the range does not include 32, the device cannot ever be Tier-1; remove from whitelist.
3. If the device claims subgroupSize=32 but the probe kernel reads a different value: file a driver bug AND temporarily push the device to Tier-2 via a runtime driver-version check at M4 probe.
4. If a real Mali run at M29 shows correctness divergence from lavapipe Tier-2: lavapipe missed a subgroup-boundary semantic; fix the Tier-2 emulation to never assume specific subgroup boundaries (use workgroup-wide `barrier()` exclusively, never `subgroupBarrier()`).
5. Update `docs/vulkan_port_architecture.md` §3 device matrix with the new Tier-1/Tier-2 line; update CHANGELOG.
6. If M29 cannot reach a real Mali or Adreno phone by release date: invoke cutline §7 — defer the missing lane to v1.1; ship v1.0-alpha with the available mobile lane validated; document the unsupported vendor's status-string disclosure (`"Adreno unsupported in v1.0-alpha; v1.1 target"` or `"Mali unsupported in v1.0-alpha; v1.1 target"`).

### R3: Wire-format byte-identity regression caught late

**Why it matters.** The whole project promise is "byte-identical .slz at L1..L5 between CUDA and Vulkan". A regression caught at M27 (full conformance gate) is enormously expensive to root-cause because dozens of kernels are now in the pipeline. A regression caught at M14 (scan_parse, the first byte-identity tripwire) is cheap.

**Detection.** M3 conformance harness lands at milestone 3 with 70 / 350 sanity baseline. M3a adds CUDA-side per-kernel intermediate dump instrumentation + per-stage golden blobs (`tests/golden_stage/`) so every per-kernel stage assertion (M11-M27) has an actual reference to compare against — the draft's per-stage gates were physically untestable without M3a. M14 (scan_parse) is the FIRST production tripwire with HARD-FAIL, per-case diff emission to `tests/diffs/{case}_L{n}_scan_parse.diff` in the documented format `OFFSET CUDA_BYTES VK_BYTES STAGE_FIELD_NAME`, and a documented triage protocol with 3-day escalation.

**Playbook if triggered.**
1. The kernel that introduced the divergence is the kernel currently being ported — diff against the previous milestone's output to confirm.
2. Categorize the divergence: (a) endianness (3-byte big-endian header read with wrong byte order), (b) alignment (off16/off32 boundary), (c) wire-format constant mismatch (HUFF_NUM_STREAMS, magic, version, codec) — caught earlier by M9's cross-language SpecConstant check, (d) sc_group_size float-vs-integer (R10 leak — should be impossible because M9's `tools/audit_spv.py` enforces zero `OpTypeFloat` in any committed SPV; if it occurs, that's a regression of M9's audit, not a new R10 occurrence), (e) match_any emulation produces different lane election (R1 correctness bug, not perf — validate against M5's standalone benchmark; CANNOT be a "site #2 semantic mismatch" because M0's site #2 rewrite makes both backends execute identical code paths), (f) shared-mem atomicAdd order divergence (R11 — but should be commutative-safe, so this would be a true bug).
3. Most divergences are (a) or (b) — fix in the GLSL source, regenerate SPV, re-run.
4. If divergence is (e), validate against M5's standalone benchmark output — does it agree? If not, the emulation regressed; if yes, the integration in the kernel introduced a bug.
5. Do not advance to the next kernel milestone until the diff is clean.
6. **Escalation timeout per plan §6**: if 3 working days of fix attempts don't close any failure mode at M14 (the FIRST byte-identity gate), file `streamlz-vk/issues/scan-parse-blocker` and pause M15-M27; consider hard scope-cut to ship v1.0-alpha with VK decode disabled.
7. If divergence is found at lane F (real mobile) or lane G (real CDNA) at M29/M30: invoke cutline §7 — fix or ship without the affected lane with documented status-string disclosure.
8. **Regression-window CI gating (M21)**: PRs touching `src_vulkan/shaders/common/*.glsl` or `src_vulkan/host/decode/*.zig` MUST re-run the Wave-1 decode gate before merge, preventing a shared-helper change from silently breaking decode while encode is the focus.

### R4: BDA absent breaks `void*` device-pointer ABI

**Why it matters.** The CUDA C ABI passes raw `void*` device pointers. Tier-1 Vulkan can use `VK_KHR_buffer_device_address` to read those pointers directly. Tier-2 Vulkan must use descriptor-set bindings, which requires the caller to register the buffer before use — and to unregister it on destruction. The architecture §13 mandate is a 16-symbol ABI (14 sibling mirrors + register + unregister), NOT the draft's claimed 15.

**Detection.** M2 ships BOTH `slzRegisterBuffer_vk` AND `slzUnregisterBuffer_vk` as no-op stubs returning SLZ_SUCCESS from day 1, so the ABI is parity from milestone 2. M6 validates BDA round-trip on every Tier-1 vendor (NVIDIA, AMD, Intel) via a small `buffer_reference` deref test AND validates the architecture §13 lifetime contract on Tier-2 (register, unregister, confirm subsequent `_vk` call returns `SLZ_ERROR_INVALID_ARG`).

**Playbook if triggered.**
1. If M6 finds BDA broken on a Tier-1 whitelisted vendor (e.g., a specific driver version returns garbage from `buffer_reference` deref): file a driver bug.
2. Pin minimum driver version in `docs/vulkan_port_architecture.md` §4 (extensions section).
3. Document in CHANGELOG which driver versions are supported.
4. If necessary, push the broken driver to Tier-2 via a runtime driver-version check at M4 probe — Tier-2 doesn't use BDA, so the broken driver path is avoided.
5. If the broken driver is widely deployed (e.g., a recent NVIDIA driver): consider shipping with a `SLZ_VK_DISABLE_BDA=1` env var that forces Tier-2 path on Tier-1-capable hardware.
6. If M6 finds Tier-2 register/unregister lifetime contract broken (e.g., post-unregister access doesn't return `SLZ_ERROR_INVALID_ARG`): the per-handle reverse-lookup pruning has a bug; fix and add a regression test.

### R5: Encode hash table > 2 GiB on single VkBuffer

**Why it matters.** Vulkan caps `VkBufferCreateInfo.size` at `maxStorageBufferRange`, which is 2 GiB on many GPUs. L5 encode on a 24 GB input produces a ~3.5 GiB hash table — won't fit in one buffer.

**Detection.** M7 stress test allocates 3 × 2 GiB hash buffers on a 24 GB-VRAM device with a synthetic kernel that BDA-addresses across buffer boundaries via a push-constant pointer table. Gated to "skipped with INFO log" on <8 GB devices (developer laptops) but always runs on a CI lane with sufficient VRAM. Adreno minStorageBufferOffsetAlignment runtime probe and maxMemoryAllocationCount enforcement land in M7 too (architecture §5 mandates). Tier-2 worst-case descriptor layout and slzCreate_vk-time hash-size decision are M22b territory because they're consumed in production lz_encode.

**Playbook if triggered.**
1. If M7 stress test fails on a 24 GB device: investigate whether the bin-pack math correctly avoids cross-buffer sub-chunks; verify the descriptor-set layout supports the worst-case N=8 buffer count on Tier-2.
2. If the failure is in the BDA push-constant table (Tier-1): push-constant size limit is 128 B per arch §4; verify the table fits in 128 B (8 × 16 B = 128 B for 8 buffers; tight).
3. If a real workload at M22b (lz_encode L5) exceeds the bin-pack ceiling: clamp the encode level on Tier-2 (effective_level_out reported via `reserved[0]`); surface via stderr per arch §6.
4. Update `docs/vulkan_port_architecture.md` §6 split math with the failure case + workaround.
5. If Adreno minStorageBufferOffsetAlignment > 4 KiB OR maxMemoryAllocationCount < 64 on a candidate Tier-2 device: rip-cord per plan §6 — reject the device with `SLZ_ERROR_VK_FEATURE_MISSING` at slzCreate_vk, document in CHANGELOG that the device is unsupported in v1.0-alpha, file v1.1 issue for a sparser allocation strategy.
