## Per-area summaries

* **cli** — The CLI split is generally clean: header comments, import order, and per-mode `run(allocator, io, w, args)` signature are consistent across the six handler files, and util.zig holds the right kind of shared plumbing. The main smells are a few real bugs hidden in bench paths (`-db -r 0` divides by zero; nonzero_kerns hard-caps at 256 silently); an asymmetric encoder/decoder driver-module alias (`gpu_encoder` vs `gpu_driver`); two different error-exit shapes in use (defensive `catch {}` in util.zig vs `try w.print` in the per-mode files); a `MemInfo.peak_rss_mb` field that is populated but never printed; and several `pub` items in util.zig that are not used outside the file.
* **decode-dispatch** — The phase extraction is genuinely clean: `FrameLayout` is a tight three-field bundle, `uploadInputAndPrefixSum` and `runBackHalf` are each self-contained, and the orchestrator reads as a linear five-step pipeline. The real smells are concentrated in the SLZ_E2E_TIMER plumbing, which has a stale dual-assignment and an edge where one cumulative timestamp is left at zero, plus a handful of small naming / parameter inconsistencies in `runBackHalf`, `mergeHuffDescs`, and `gatherRawOff16`. None are correctness bugs against real frames; they would mostly trip up the next person reading these files.
* **decode-rest** — Two real correctness defects survived the cleanup: `d_entropy_off16_scratch` and `d_first_subchunk_idx` are documented as owned device buffers, but the dispatch only ever assigns them as views into other buffers — yet `DecodeContext.deinit` calls `cuMemFree` on them. The result is a silent CUDA error (interior pointer) plus a silent double-free of `d_first_sub_idx_persist`'s allocation on every per-handle teardown. The rest of the area is mostly clean refactor — the smells that remain are stale doc references (dangling `walkResultToHost` / `walkMetaToHostAsync`, wrong `src/gpu/...` paths, reversed v1/v2 wording on `UnsupportedVersion`, the `uint4` justification for `ChunkDesc.reserved` that no longer makes sense at 24 bytes), one inconsistency where `block_header.zig` claims the decoder accepts `.high` but `streamlz_decoder.dispatchCompressedBlock` rejects it, and the now-dead defensive `pipeline_stream_created` check at decode_dispatch:706 because `ensurePipelineStreams` was hoisted to line 650.
* **encode-pipeline** — The encode pipeline post-refactor is mostly clean. The `walkStream` / `encodeStreamAndPublish` collapse in `encode_huff.zig` is solid for the lit/tok path, but the helper boundary leaks `offsets` ownership asymmetrically and the off16 wrapper has a real allocation-leak bug introduced by the rewrite. `fast_framed.zig` carries a couple of stale roadmap labels and an orphaned docstring above `gpuFrameAssembleImpl`. The four `freeHuff*` helpers and `freeAssembled` are a near-perfect candidate for a single shared free helper.
* **encode-rest** — The encode-rest surface is mostly clean, but several docstrings still describe the pre-refactor world. The biggest concern is the `assembled_offsets`/`assembled_sizes` field doc in `encode_context.zig`: it claims the tables are per-chunk, while `gpuAssembleFrameImpl` actually publishes them per sub-chunk — and a stale "OWNERSHIP RULE" reference in `EncodeContext.deinit` points at a contract that no longer exists. `levels.zig` and the two FFI/loader files still document a CPU-engine path that has been deleted, and `cuda_ffi.zig` / `module_loader.zig` justify the encode-side FFI shim with a factually wrong claim about which entries decode resolves.
* **c-abi** — The C ABI surface is in good shape after the cleanup: the error-mapping switches enumerate every upstream variant explicitly, the comptime block locks the option-struct ABI offsets, and the device-only sentinel is well-documented. Findings are mostly doc/impl drift and one write-only helper. No correctness bugs found in the size-guard math, deinit ordering, or `version.string` plumbing.
* **format-and-tree** — Format module is tight but carries several real bugs from the v1→v2 rewrite: ParseError docs lie about what each error means (UnsupportedVersion still says "is not 1", BadCodec says "only `.fast`", BadScGroupSize claims a > 4.0 check that doesn't exist). * **deferral-defender** — Two of the four deferrals are defensible (DevBuf wrapper, level default mismatch).
